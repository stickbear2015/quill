# QUILL Startup Wizard — Design Plan

**Purpose:** First-run (and re-run) wizard that lets users shape QUILL into their editor.
Features they disable are completely hidden — no menus, no commands, no noise.
**Audience:** This document is the authoritative implementation spec.

---

## 1. Goals

1. Every user gets a QUILL that fits their work and accessibility needs from minute one.
2. Disabled features disappear from menus, the command palette, and keyboard shortcuts.
3. The wizard is accessible by keyboard and screen reader from the first keystroke.
4. Re-running the wizard at any time is a single command.
5. No feature is permanently locked out; any choice can be reversed.

---

## 2. High-Level Architecture

```
quill/
  ui/
    setup_wizard.py          -- wx.adv.Wizard subclass; all pages
    setup_wizard_pages.py    -- individual WizardPage classes
  core/
    feature_flags.py         -- FeatureFlags dataclass, load/save helpers
    feature_registry.py      -- command-to-feature-flag mapping table
  tools/
    check_feature_tags.py    -- CI gate: every command must declare its feature tag
```

The wizard writes a `feature_flags` block into `settings.json` via the existing
`write_json_atomic` path. On startup `MainFrame.__init__` loads flags before
building menus, the command registry, and keybindings.

---

## 3. Feature Flag Schema

New block added to `settings.json` (validated by existing schema infrastructure):

```json
{
  "setup_wizard_completed": true,
  "feature_flags": {
    "ai":           true,
    "remote":       false,
    "quillins":     true,
    "power_tools":  true,
    "notebook":     false,
    "collaboration": true
  }
}
```

`FeatureFlags` is a frozen dataclass in `quill/core/feature_flags.py`.
`is_enabled(flag: str) -> bool` is the single access point everywhere.

When `setup_wizard_completed` is absent or `False`, the wizard runs at startup
before `MainFrame` is shown.

---

## 4. Wizard Pages — Full UX Specification

The wizard is a `wx.adv.Wizard`. Each page is a `wx.adv.WizardPageSimple`.
Navigation: **Back**, **Next / Finish**, **Cancel** (with confirmation).
All heading text is read as a live region when the page loads so screen readers
announce the page title without user action.

---

### Page 1 — Welcome

**Heading:** "Welcome to QUILL"

**Body text (read as static text, no heading markup so it flows naturally under
the page title):**

> QUILL is a screen-reader-first writing editor. This two-minute setup wizard
> personalises your copy so that only the features you want appear in the menus
> and command palette.
>
> You can change any of these choices later from the Tools menu — just run
> "Personalise QUILL" to come back here.
>
> Press Next to begin.

**Controls:** Next button only (no Back on page 1). Cancel asks "Exit without
setting up? QUILL will run with default settings." with Yes / No.

**Accessibility notes:**
- Focus lands on the Next button when the page loads.
- The static text block has `SetName("Welcome message")` so a screen reader
  user who tabs backward to it hears a meaningful label.

---

### Page 2 — Your Screen Reader

**Heading:** "Accessibility"

**Purpose:** Set the screen-reader profile up front so that all subsequent
pages use correct announcement strategies.

**Question 1 — "Are you using a screen reader?"**
Radio group (vertical, labelled "Screen reader use"):
- Yes, I use NVDA
- Yes, I use JAWS
- Yes, I use another screen reader
- No, I don't use a screen reader

Selecting any "Yes" option enables full Prism/pyttsx3 announcement mode and
sets `accessibility_mode = "screen_reader"` in settings.

**Question 2 — "QUILL key" (visible for all users)**
A `wx.Choice` labelled "QUILL key (used for editor commands)":
- Caps Lock (default for screen reader users)
- Insert (JAWS/NVDA users who prefer the numpad)
- None (sighted users who don't want a QUILL key)

Explanation below the choice (static text, updates dynamically as choice
changes): "The QUILL key is a dedicated modifier that unlocks reading and
navigation commands. You can always remap it in Settings."

**Question 3 — "Sound effects"**
A single `wx.CheckBox` labelled "Play sound effects for actions (open, save,
alerts)". Checked by default.

**Accessibility notes for this page:**
- When a radio button is selected the dynamic explanation below it is announced
  via `wx.adv.NotificationMessage` or a focused static text region.
- The QUILL key choice fires `EVT_CHOICE` and immediately updates the explanation
  so the user hears what that key will do before pressing Next.

---

### Page 3 — AI Writing Assistance

**Heading:** "AI Writing Assistance"

**Gate decision — radio group "Would you like AI writing assistance?":**
- Yes, connect to an AI provider
- No, I'll write without AI

If **No** is selected:
- The rest of this page collapses (all child controls hidden, sizer re-laid out).
- `feature_flags.ai` is set to `False`.
- The page announces: "AI features will be hidden. You can enable them later
  from Settings."

If **Yes** is selected, three sub-sections become visible:

**Sub-section A — Provider**
`wx.Choice` labelled "AI provider":
- OpenAI  (cloud, requires API key)
- OpenRouter  (cloud, many models, requires API key)
- Ollama  (local, free, requires Ollama running on your machine)
- Set up later

When the choice changes, sub-sections B and C update immediately (same
per-provider model memory pattern already used in the Ask Quill dialog).
The announcement reads: "Selected [provider]. [Brief one-line description]."

**Sub-section B — API Key** (shown only for OpenAI / OpenRouter)
`wx.TextCtrl(style=wx.TE_PASSWORD)` labelled "API key".
Below it: a `wx.Button` labelled "Reveal key" toggles `wx.TE_PASSWORD`.
Below the button: a static text hint "Your key is stored in Windows Credential
Manager, never in any file."

**Sub-section C — Local host** (shown only for Ollama)
`wx.TextCtrl` labelled "Ollama host" with default value `http://localhost:11434`.
Static text hint: "Make sure Ollama is running before pressing Next."

**Sub-section D — Default model**
`wx.TextCtrl` labelled "Default model". Pre-filled with the provider's
recommended default (same `default_model_for_provider()` logic used elsewhere).
The field is always shown when a real provider is selected; hidden for
"Set up later".

**Accessibility notes for this page:**
- Hiding/showing sub-sections uses `sizer.Show(ctrl, show)` + `Layout()` so the
  tab order is always correct and no hidden control receives focus.
- The live region for the provider choice change announcement is the static
  description text immediately below the choice, which is re-populated and
  then `SetFocus()`-ed for one keystroke so NVDA/JAWS reads it, then focus
  returns to the choice.

---

### Page 4 — Remote File Editing

**Heading:** "Remote File Editing"

**Gate decision — radio group "Do you edit files on remote servers?":**
- Yes, I use SSH or SFTP
- No, I work with local files only

If **No**: `feature_flags.remote = False`. Page collapses to a single
confirmation line: "Remote editing will be hidden from menus."

If **Yes**:
- Static text: "QUILL can open, edit, and save files directly over SSH/SFTP.
  You can add saved sites now or later from the Remote menu."
- `wx.CheckBox` labelled "Add a saved site now". Default: unchecked.
- When checked, a mini site-entry form (Friendly name, Host, Port, Username)
  appears inline. This is not mandatory — the user can add sites later from
  the Remote menu or via Manage Remote Sites.
- The mini form uses the same factory-lambda `row()` pattern as the full
  `SiteEditDialog` so z-order is correct from the start.

**Accessibility notes:**
- The inline site form appears/disappears based on the checkbox via the same
  show/hide sizer pattern as Page 3.
- Tabbing into the collapsed area is impossible (no hidden focusable controls).

---

### Page 5 — QUILL Extensions (Quillins)

**Heading:** "Extensions"

**Explanation (static text):**
> QUILL extensions (Quillins) add capabilities like word count indicators,
> custom shortcuts, AI writing prompts, and more. Bundled extensions include
> an AI Prompt Pack and a Word Count sidebar.

**Gate decision — radio group "Would you like to enable extensions?":**
- Yes, enable extensions
- No, keep it simple

If **No**: `feature_flags.quillins = False`. Extensions menu and all
extension-contributed commands are hidden.

If **Yes**:
- `wx.CheckBox` labelled "Auto-install bundled extensions" (checked by default).
  If checked, the bundled Quillins under `quill/quillins_bundled/` are enabled.
- Static list (read-only `wx.ListCtrl` with two columns: Name, Description)
  shows the bundled extensions so the user knows what they are getting.
  Screen reader users can navigate the list with arrow keys; each row reads
  "Name. Description."

---

### Page 6 — Power Tools

**Heading:** "Power Tools"

**Explanation (static text):**
> Power Tools are advanced text operations: sort lines, shuffle text, strip
> HTML, encode/decode entities, search across files, and more. They live in
> the Power Tools menu and the command palette.

**Gate decision — radio group "Include Power Tools?":**
- Yes, I want these tools
- No, keep the menus focused

If **No**: `feature_flags.power_tools = False`. Power Tools menu and all
power-tool commands are hidden.

This page is short by design. After the radio selection there is nothing else
to configure; the user presses Next immediately.

---

### Page 7 — Notebook Workspace

**Heading:** "Notebook Workspace"

**Explanation (static text):**
> The Notebook workspace lets you work on a collection of related documents
> together — a research notebook, a book project, or a set of notes. Each
> document in the notebook can link to others.

**Gate decision — radio group "Enable the Notebook workspace?":**
- Yes, I work on multi-document projects
- No, I work one document at a time

If **No**: `feature_flags.notebook = False`. The Notebook menu entry and
all Notebook commands are hidden.

---

### Page 8 — Keyboard Profile

**Heading:** "Keyboard Shortcuts"

**Explanation (static text):**
> Choose a starting keyboard profile. You can fine-tune individual bindings
> at any time from Settings > Keyboard.

**Radio group "Keyboard profile":**
- QUILL Default — all built-in bindings active
- Minimal — only File, Edit, and Format shortcuts; all advanced chords off
- Screen Reader Friendly — remaps conflicts with common NVDA/JAWS shortcuts

Each radio button, when focused, triggers a brief description line below:
"[Profile name]: [one sentence description of what changes]."

The profile is stored as `keymap_profile` in settings and applied by the
keymap loader at startup. Profiles are JSON files in `quill/core/keymap/`.
The wizard page lists them by scanning that directory, so future profiles
appear automatically.

**Accessibility notes:**
- The description update on focus (not just on selection) means arrow-key
  browsing of the radio group gives immediate auditory feedback about each
  option before committing.

---

### Page 9 — Summary and Finish

**Heading:** "Your QUILL Setup"

**Purpose:** Give the user a clear, auditable summary of what they chose
before anything is written to disk.

**Body:** A `wx.ListCtrl` (report view, two columns: Feature, Status) listing
every decision made in pages 2-8:

| Feature              | Status         |
|----------------------|----------------|
| Screen reader        | NVDA           |
| QUILL key            | Caps Lock      |
| Sound effects        | On             |
| AI assistance        | OpenAI         |
| Remote editing       | Disabled       |
| Extensions           | Enabled        |
| Power Tools          | Enabled        |
| Notebook workspace   | Disabled       |
| Keyboard profile     | QUILL Default  |

The list is read-only and navigable by arrow keys. Each row announces
"Feature: [name]. Status: [value]." when focused.

Below the list, a single static text line:
"Press Finish to apply these settings and open QUILL."

**Back button** returns to Page 8 so the user can change anything.
**Finish button** (replaces Next on the last page) writes settings and
closes the wizard. `setup_wizard_completed = True` is written atomically.

After the wizard closes, `MainFrame._apply_feature_flags()` is called to
rebuild menus, refresh the command palette, and re-bind the keymap. If this
is first-run, QUILL then shows an empty document with the cursor ready.

## 5. Feature Flag Enforcement

### 5.1 Menu Filtering

`main_frame_menu.py` calls `is_enabled(flag)` before appending each feature-
group's menu items. Items are not created at all (not just hidden) when their
flag is off. This keeps the menu object model clean and ensures screen readers
see no phantom items.

Each menu item has a `_FEATURE_TAG` constant so future developers can see
at a glance which flag governs it.

```python
# Example pattern in main_frame_menu.py
if flags.ai:
    menu.Append(_id_ask_ai, "Ask &Quill\tAlt+Q")
    menu.Append(_id_prompt_library, "Prompt &Library")
    menu.Append(_id_skill_library, "Skill &Library")
    # ... etc
```

### 5.2 Command Palette Filtering

Every command in the `CommandRegistry` gains a `feature_tag: str | None`
field. The palette's search backend calls `is_enabled(cmd.feature_tag)` and
excludes commands whose tag is disabled. Commands with `feature_tag = None`
are always shown (core editing commands).

Feature tag constants are defined in `feature_flags.py`:

```python
FEATURE_AI        = "ai"
FEATURE_REMOTE    = "remote"
FEATURE_QUILLINS  = "quillins"
FEATURE_POWER     = "power_tools"
FEATURE_NOTEBOOK  = "notebook"
```

### 5.3 Keyboard Binding Filtering

The keymap loader skips binding registrations for commands whose feature tag
is disabled. This prevents ghost shortcuts that do nothing from cluttering
`Ctrl+?` discovery output.

### 5.4 CI Gate — `check_feature_tags.py`

A new gate tool (same family as `check_dialog_zorder.py`) walks the command
registry at import time and asserts that every command either has a known
`feature_tag` value or explicitly declares `feature_tag=None`.

Run: `python -m quill.tools.check_feature_tags`
Test: `tests/unit/tools/test_feature_tags.py`

## 6. First-Run Detection and Re-run

### First run

In `MainFrame.__init__`, before any UI is shown:

```python
if not load_settings().setup_wizard_completed:
    from quill.ui.setup_wizard import run_setup_wizard
    run_setup_wizard(parent=None)  # modal, no parent yet
    # settings are now written; reload before building menus
    reload_settings()
```

The wizard window is created as a top-level frame (not a child of MainFrame)
so it can be the application's main window during the setup phase.

### Re-run

`Tools > Personalise QUILL...` (command `open_setup_wizard`, always visible
regardless of flags, feature_tag=None). This opens the wizard in update mode:
all pages are pre-filled with current settings. On Finish, changed flags
trigger `_apply_feature_flags()` to rebuild the affected menu groups and
refresh the palette without a restart.

If re-run disables a feature that was previously enabled, a one-time
announcement reads: "AI features have been hidden. You can re-enable them
here at any time."

---

## 7. `_apply_feature_flags()` — Live Rebuild

This method on `MainFrame`:

1. Calls `_rebuild_menus()` (already exists for dynamic menu changes).
2. Calls `command_registry.rebuild_filtered_set(flags)`.
3. Calls `keymap.apply_profile(settings.keymap_profile)`.
4. Calls `announce("Setup complete. [N] features enabled, [M] hidden.")`.
5. Calls `Layout()` on the frame.

All of this is done on the UI thread (it is synchronous and fast).

---

## 8. Accessibility Requirements for the Wizard Itself

1. Every page announces its heading via a live region the moment it becomes
   active (use `wx.adv.Wizard.EVT_WIZARD_PAGE_CHANGED`).
2. Focus on page change lands on the first interactive control (or the Next
   button if the page has no inputs).
3. Show/hide of conditional sub-sections (AI provider details, inline site
   form) must not leave focus on a now-hidden control. After hiding, focus
   moves to the Next button; after showing, focus moves to the first new
   control.
4. All radio groups are wrapped in a `wx.StaticBox` labelled with the question
   text. NVDA reads this as a group name, giving full context.
5. Error states (e.g. blank API key when OpenAI is selected and the user tries
   to advance) use a `wx.adv.NotificationMessage` balloon anchored to the
   offending field. The balloon text is also set as `SetToolTip` on the field
   so it is keyboard-accessible.
6. The summary list on Page 9 uses `wx.LC_REPORT` with explicit column headers
   ("Feature" and "Status") so JAWS and NVDA announce the column name with
   each cell when reading in table mode.
7. Cancel confirmation uses `wx.MessageDialog` (not a raw `ShowModal`) so the
   dialog contract is satisfied and keyboard shortcuts (Enter/Escape) work
   correctly.

## 9. Settings Spec Additions

New fields in `settings_specs.py`:

| Field                           | Type    | Default | Description                           |
|---------------------------------|---------|---------|---------------------------------------|
| `setup_wizard_completed`        | bool    | False   | Suppresses wizard on next launch      |
| `feature_flags_ai`              | bool    | True    | Master switch for all AI features     |
| `feature_flags_remote`          | bool    | True    | Master switch for remote editing      |
| `feature_flags_quillins`        | bool    | True    | Master switch for Quillins            |
| `feature_flags_power_tools`     | bool    | True    | Master switch for Power Tools         |
| `feature_flags_notebook`        | bool    | True    | Master switch for Notebook workspace  |
| `keymap_profile`                | str     | "default" | Active keyboard profile name        |
| `accessibility_mode`            | str     | "auto"  | "screen_reader", "visual", or "auto"  |

Flat fields (not nested JSON) to remain consistent with the existing
single-level settings schema and DPAPI encryption surface.

## 12. Implementation Phases

### Phase 1 — Feature Flag Infrastructure (no wizard yet)

1. Add `feature_flags.py` with `FeatureFlags` dataclass.
2. Add settings spec fields; update `Settings` dataclass.
3. Wrap menu groups in `main_frame_menu.py` and
   `main_frame_power_tools_menu.py` with flag guards.
4. Add `feature_tag` to `Command`; update command registrations in
   `main_frame.py` (one sweep to tag all commands).
5. Implement `rebuild_filtered_set` on registry; wire into command palette.
6. Add `check_feature_tags.py` gate and test.
7. **Milestone:** Flag defaults are all `True`; everything works as before.
   Test by manually toggling flags in `settings.json`.

### Phase 2 — Wizard UI

1. Implement `setup_wizard_pages.py` (all 9 page classes), starting with
   pages 1, 2, 9 (welcome, accessibility, summary) as scaffolding.
2. Add pages 3-8 one at a time, each followed by a test run.
3. Implement `setup_wizard.py` wiring all pages together with Back/Next logic.
4. Write `tests/unit/ui/test_setup_wizard.py` covering:
   - Page-change event fires heading announcement.
   - Selecting "No" on AI collapses sub-sections.
   - Summary list reflects all choices correctly.
   - Cancel confirmation dialog appears.
5. Add `run_setup_wizard()` public entry point.

### Phase 3 — First-Run and Re-run Integration

1. Wire first-run check into `MainFrame.__init__`.
2. Implement `_apply_feature_flags()` on `MainFrame`.
3. Add "Personalise QUILL..." to Tools menu (always visible).
4. End-to-end test: fresh settings file, wizard runs, flags applied, menus
   reflect choices.

### Phase 4 — Keymap Profiles

1. Extract current default bindings into `profile_default.json`.
2. Author `profile_minimal.json` and `profile_sr_friendly.json`.
3. Update keymap loader to accept a profile name.
4. Wire profile choice on wizard Page 8 to the loader.

### Phase 5 — Polish and Documentation

1. Screen-reader announcement audit (all pages, all branches).
2. Tab order verification with NVDA and JAWS.
3. Update `docs/userguide.md` with "First Launch" section.
4. Update `docs/QUILL-PRD.md` §12.

## 13. Acceptance Criteria

- [ ] Wizard runs automatically on first launch (no `setup_wizard_completed`).
- [ ] Selecting "No" for AI removes: Ask Quill, Prompt Library, Skill Library,
      grammar check, document summary from menus AND command palette.
- [ ] Selecting "No" for Remote removes: SSH, FTP, Remote Sites from menus.
- [ ] Selecting "No" for Power Tools removes the Power Tools menu entirely.
- [ ] Selecting "No" for Notebook removes the Notebook menu entry.
- [ ] "Personalise QUILL..." re-runs wizard; changes apply without restart.
- [ ] All wizard pages pass keyboard-only navigation (no mouse required).
- [ ] NVDA announces each page heading on page load.
- [ ] NVDA announces show/hide of conditional controls correctly.
- [ ] Cancel from page 1 leaves settings unchanged.
- [ ] `check_feature_tags` gate is green in CI.
- [ ] Full test suite passes (`pytest tests/unit tests/stability`).
