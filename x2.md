## Resolved findings (moved here from x2.md)

### #1 — ts_console._handle_invoke ran wx widget calls on the Node reader thread

**Solution:** Added a `ui_call: Callable[[Callable[[], Any]], Any] | None` parameter to
`TypeScriptConsole.__init__`. When set, `_dispatch` marshals each invoke execution
through it — blocking the reader thread on a `threading.Event` until the UI thread
completes the call. In `DevToolsMixin._dt_ts_console()`, the console is constructed
with `ui_call=self._dt_marshal_to_ui`. `_dt_marshal_to_ui` posts a wrapper to
`wx.CallAfter`, waits on an `Event`, and re-raises any exception. The UI thread
is free during invoke processing because `execute()` runs on `_dt_ts_execute_thread`,
not the UI thread.

**Impact:** Eliminates undefined behavior: all `ConsoleHost` wx widget access
(`editor.GetValue()`, `WriteText`, etc.) now runs on the UI thread as required.
No deadlock risk — the design was already correct (execute on daemon thread,
UI thread free), just missing the marshal step.

### #2 — restart_typescript_worker created wx.Frame off UI thread

**Solution:** Added a guard at the top of `restart_typescript_worker()`:
if `_dev_console_window` has never been created (`not hasattr(self, "_dev_console_window")`),
announce an actionable message and return. Previously the method created a `ConsoleWindow`
from a daemon thread via `self._dt_window()` in the `CallAfter` argument list, violating
the wx threading invariant.

**Impact:** Eliminates potential crash from wx widget creation off the UI thread.
Users who invoke the menu item before ever opening the console get a clear message.

### #23 — github-temp filename collision caused wrong-repo commits

**Solution:** In `_on_file_fetched`, replaced the bare `<basename>` local path with a
content-addressed subdirectory keyed on `(repo, ref, path)`:
```python
slot_key = f"{remote_file.repository.full_name}:{remote_file.ref}:{remote_file.path}"
slot_hash = hashlib.sha256(slot_key.encode()).hexdigest()[:16]
slot_dir = temp_dir / slot_hash
local_path = slot_dir / Path(remote_file.path).name
```
Each unique `(repo, ref, path)` combination now lands in its own slot, so the
`origins` dict entry is never overwritten by a same-named file from a different repo.

**Impact:** Opening `README.md` from two different repositories no longer causes
Tab A's "Save to GitHub" to silently commit to Tab B's origin. The collision that
could corrupt data across tabs is eliminated.

**Solution:** Added `_looks_binary(raw: bytes) -> bool` to
`quill/core/github/github_provider.py` — a NUL-byte heuristic that's correct
for the dominant binary formats (PNG, ZIP, JPEG, ELF, etc.) without producing
false positives on legitimate UTF-8 source. `get_file()` now raises

the announce-dialog wrapper (`_show_message_box`) with an actionable message

to inspect or modify it.") instead of silently substituting a latin-1
fallback.

**Impact:** Opening a PNG from GitHub no longer writes garbled bytes back to
the repo on Save. A "binary file" message is announced to the screen reader
so the user knows why the open failed. The original UTF-8 text path is
untouched.

Tests: `tests/unit/core/test_github_provider.py` (6 tests covering

and the latin-1 fallback path).

### #32 — Downloaded GitHub files persisted in app-data/github-temp indefinitely

**Solution:** Added `prune_orphaned_github_temp()` to `GitHubRemoteMixin`.
It walks `app_data_dir() / "github-temp"`, computes the set of paths the
mixin's `_gh_state().origins` dict still claims, and deletes the on-disk
files whose content-addressed slot hash is no longer represented in
`origins`. The prune is wired into `MainFrame._on_close` between
`close_ssh_connections()` and `save_settings()`, wrapped in
`getattr(..., None)` and a try/except so a prune failure can never block
shutdown.

**Impact:** Every QUILL session now reaps the github-temp files that are no
longer backed by an open tab. Users who browse dozens of files in one
session don't accumulate megabytes of private repo content on disk forever.
PRIVACY.md continues to document the temp directory for manual cleanup.

### #35 — provider.close() existed but no caller ever closed providers

**Solution:** Wrapped every provider construction in `quill/ui/main_frame_github.py`
with `contextlib.closing(GitHubRemoteProvider(token=...))` and changed every
worker thread to consume the context manager:
```python
with contextlib.closing(GitHubRemoteProvider(token=token or None)) as provider:
    self._set_status(...)
    threading.Thread(  # GATE-40-OK: ...
        target=self._worker,
        args=(provider, ...),
    ).start()
```
…and inside the worker:
```python
with provider as live_provider:
    result = live_provider.get_file(...)
```
This applies to all five call sites: `open_github_repository`,
`open_github_file_url`, the identity probe in `manage_github_accounts`,
`_open_worker`, and `_save_back_worker`. The GitHub session is now
guaranteed to be released on the worker thread, even if the user quits
mid-save.

**Impact:** Eliminated the per-operation leaked `Github` instance. The
session pool that backs PyGithub can no longer be exhausted by a heavy
browse session. Quitting QUILL mid-fetch no longer leaves a session
hanging in the pool.

### #36 — fetch_bootstrappers.py downloaded bootstrap.exe from unpinned HEAD

**Solution:** Three coordinated changes in `scripts/fetch_bootstrappers.py`:

1. Added a `DEFAULT_BASE_URL` constant pinned to a specific commit SHA of
   `AccessibleApps/app_updater`. The placeholder
   `<REPLACE_WITH_COMMIT_SHA>` is replaced at release time, and
   `_resolve_base_url()` raises `ValueError` if the placeholder is still
   present so a release cannot ship without the SHA being filled in.
2. Added `_resolve_base_url()` which explicitly refuses `HEAD`, `master`,
   `main`, and any ref that is not a 40-character commit SHA. The function
   honors `QUILL_BOOTSTRAPPER_BASE_URL` env var and `--base-url` CLI arg
   for development overrides.
3. Threaded `base_url` through `_download()` so every fetch routes through
   the resolved pinned URL.

**Impact:** A future compromise of the third-party `app_updater` repo can
no longer be pushed to QUILL users at release time. The bootstrapper is
fetched from a SHA that was reviewed and committed by the release
engineer. CI can additionally run `--verify-only` against the committed
`checksums.sha256` to fail a release on any drift.

Tests: `tests/unit/scripts/test_fetch_bootstrappers.py` (7 tests covering
SHA validation, HEAD/master/main rejection, env var override, CLI arg
override, and the placeholder refusal).
